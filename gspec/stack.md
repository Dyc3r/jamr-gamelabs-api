# Technology Stack

## 1. Overview

- **Architecture style:** Monolith — a single FastAPI service exposing a REST API. No microservices split; scale and consumer count (one internal portal application) don't warrant it.
- **Deployment target:** Docker container(s) on a single host/VM. No container orchestration platform. This repo owns only its own application container definition — backing services (Postgres, self-hosted Zitadel) are provided by a parent project's docker-compose setup that imports this container.
- **Scale and performance requirements:** Low-to-moderate. Single known consumer (the portal application), no public traffic, no expected high-concurrency spikes. Architecture favors simplicity and maintainability over horizontal scalability.

## 2. Clarifications

- **CI/CD platform:** Deferred. No CI/CD platform has been chosen yet. GitHub Actions is a reasonable default given where this repo is hosted, but this was left open rather than assumed.

## 3. Core Technology Stack

### Programming Languages

- No secondary languages.
- **Tooling:** Ruff (linting + formatting) and mypy (static type checking).

### Runtime Environment

- **Runtime:** CPython 3.12, running inside Docker containers (`python:3.12-slim` base image).

## 4. Frontend Stack

Not applicable — this is a headless REST API with no frontend. The API's admin capabilities are surfaced through a separate portal application's UI, which is out of scope for this stack document.

## 5. Backend Stack

### Framework

- **FastAPI** (`>=0.115`) — async-native, OpenAPI schema generation out of the box, strong Pydantic integration for request/response validation.
- **API style:** REST.
- **ASGI server:** Uvicorn (`standard` extras) — `--reload` in dev, multi-worker (`--workers 4`) in prod.

### Database

- **PostgreSQL** — required per the product profile.
- **Driver:** `psycopg3`, used directly via its native async support (`psycopg.AsyncConnection`, pooled with `psycopg_pool.AsyncConnectionPool`) — matches the rest of the stack being async throughout. Flag if sync was actually intended instead.
- **No ORM.** The data model is narrow enough that an ORM adds overhead without earning it. Queries are hand-written SQL, executed directly through `psycopg3`.
- **Migrations:** Alembic, used for migration bookkeeping (version table, ordering, up/down) — every migration is **hand-written SQL**, not autogenerated (there are no ORM models to diff against).

### Caching Layer

Not applicable for now — current scale (single internal consumer) doesn't justify one. Revisit if read-heavy endpoints (e.g., stat leaderboards) show latency pressure.

### Message Queue / Event Bus

Not applicable — the Patreon sync is a manually triggered operation via an admin-facing API endpoint. No scheduling or queuing infrastructure is required.

## 8. Authentication & Security

### Authentication

- **Zitadel (self-hosted)** — the identity provider. Zitadel is OIDC-compliant: incoming JWTs are validated against Zitadel's published JWKS. Runs as part of the parent project's shared local/deployment stack rather than Zitadel Cloud.
- **Flow:** Bearer JWT access tokens issued by Zitadel, validated per-request by this API (no session state held here).
- **Libraries:** `PyJWT` + `cryptography` for token verification, `httpx` for JWKS retrieval/caching and any Zitadel management API calls.

### Authorization

- **Pattern:** Role-based access control (RBAC), high-level only — the specific requirement from the product profile is restricting the admin dashboard's underlying endpoints to users with an admin role.
- **Enforcement:** Role claims are expected in the Zitadel-issued JWT; a FastAPI dependency validates the token and checks for the required role before allowing access to admin-scoped routes. Fine-grained permission modeling (beyond "admin vs. not") is not yet required — see `gspec/architecture.md` for where this is enforced in code.

### Security Tools

- **Secrets management:** Environment variables via `.env` (local) / container environment injection (prod). No dedicated secrets manager at this scale.
- **Error tracking doubles as a security signal:** Sentry (below) surfaces unexpected auth/validation failures.

## 14. Technology Decisions & Tradeoffs

### Key Architectural Decisions

- **No ORM — raw `psycopg3`.** The data model is narrow enough that ORM overhead isn't worth it. Alembic is retained for migration bookkeeping, but migrations are hand-written SQL.
- **`uv` for package management** — dependency resolution, lockfile, and virtualenv management in one fast tool.
- **PostgreSQL** as the primary datastore, per the product profile's explicit requirement.
- **Self-hosted Zitadel via OIDC** for authentication — JWT + JWKS validation, with role claims driving admin-dashboard RBAC.
- **This project does not own its local backing services** — no Postgres or Zitadel container in this repo's own compose setup. It defines only its application container, which the parent project's docker-compose imports alongside the shared Postgres and self-hosted Zitadel instances used by the full local stack.
- **Patreon sync is manually triggered**, not scheduled — no systemd timers, schedulers, or queue infrastructure, at the cost of requiring a human to trigger syncs.
- **Ruff + mypy** for linting, formatting, and static type checking.

### Risk Mitigation

- **Identity provider vendor risk:** mitigated by using standard OIDC — the integration is not tightly coupled to Zitadel-specific APIs beyond JWKS validation and (optionally) admin API calls.
- **No CI/CD yet:** Until a platform is chosen, rely on local `ruff`/`mypy`/`pytest` runs before merging. This is a known gap, flagged in Clarifications for deliberate follow-up.
- **Manual Patreon sync risk:** Entitlements can go stale if nobody triggers a sync. Acceptable at current scale per explicit product decision; revisit if this becomes an operational pain point.

## 15. Technology-Specific Practices

### Framework Conventions & Patterns

- FastAPI routers organized per domain module (see `gspec/architecture.md` for the concrete folder structure), each exposing an `APIRouter` included into the main app in `app/main.py`.
- Request/response schemas defined as Pydantic models. With no ORM, there are no separate persistence-layer model classes to keep them apart from — query results are mapped directly into Pydantic schemas (or plain dataclasses) at the service layer.
- Auth/RBAC enforcement via FastAPI dependencies (`Depends(...)`), not middleware, so protection is explicit per-route rather than global.

### Library Usage Patterns

- `psycopg3` usage: acquire connections from a single shared `psycopg_pool.AsyncConnectionPool`, hand-written parameterized SQL per query, using a row factory (e.g., `dict_row` or `class_row`) to map results into typed structures rather than raw tuples.
- Alembic migrations: hand-written SQL per migration (no autogeneration, since there's no ORM metadata to diff against) — one migration per meaningful schema change.
- `pydantic-settings` for all environment/config loading — no direct `os.environ` reads scattered through the codebase.

### Language Idioms

- Type hints required on all function signatures (enforced by mypy).
- Prefer `async`/`await` throughout the request path — no blocking I/O calls (e.g., synchronous `requests`, sync `psycopg3` connections) inside async route handlers or services.

### Stack-Specific Anti-Patterns

- Don't perform blocking database or HTTP calls inside async route handlers — always use the async `psycopg3` pool and `httpx.AsyncClient`.
- Don't bypass Alembic by editing the database schema directly — every schema change goes through a migration.
- Don't build SQL via string concatenation/f-strings — always use `psycopg3`'s parameterized query placeholders to avoid SQL injection.
- Don't scatter Zitadel/Patreon API calls throughout route handlers — centralize each integration behind a dedicated client/service module (see `gspec/architecture.md`) so token handling, retries, and error mapping live in one place.
